# Windtrap vanilla crash — status as of 1.0.274

## What we know

The vanilla Windtrap render-time crash (`0xC0000005 at offset 0x54F8` of
`dunecity.exe`) reproduces on every mission transition in vanilla mod (no
Tornie mod active). It is **NOT** the WindTrap class itself.

The crash signature is constant across:
- 1.0.269: `0x00007FF7BCDA54F8`
- 1.0.271: `0x00007FF7FA6054F8`
- 1.0.272: `0x00007FF72A6154F8`
- 1.0.274: `0x00007FF743E154F8`

All four have offset `0x1400454F8` in `dunecity.exe` (verified by
subtracting the `0x140000000` image base — the upper bits vary due to
ASLR randomization, but the offset within the file is constant).

## What the disassembly at offset 0x140045480 says

The function that crashes (start at `0x140045480` per `rex push %rbx;
push %rbp; push %rsi; push %rdi; lea -0x2(%rdx), %r10d; ...`) is a
nested loop:

```
140045480:  push %rbx; push %rbp; push %rsi; push %rdi
140045485:  lea -0x2(%rdx), %r10d
140045489:  mov %rcx, %r11
14004548c:  lea 0x2(%rdx), %esi
14004548f:  cmp %esi, %r10d
140045492:  jg  0x14004550c       ; exit if bounds invalid
140045494:  movzbl 0x1a(%r9), %edi   ; read byte at offset 0x1a of arg 3
140045499:  lea -0x2(%r8), %ebp
14004549d:  lea 0x2(%r8), %ebx
1400454a1:  mov (const_pool), %r8
1400454a8:  mov %ebp, %eax
1400454b0:  test %r10d, %r10d       ; inner loop bound
1400454b5:  cmp (%r11), %r10d
1400454b8:  jge 0x1400454fe
1400454ba:  test %eax, %eax
1400454bc:  js  0x1400454fe
1400454eb:  mov 0x9678(%r8, %rdx, 8), %rdx   ; INDEXED ARRAY LOAD
1400454f3:  test %rdx, %rdx
1400454f6:  je  0x1400454fe       ; nullptr check (skips successfully)
1400454f8:  cmp %dil, 0x1a(%rdx)  ; *** CRASH HERE ***
```

The crash instruction `cmp %dil, 0x1a(%rdx)` is an OOB read on a pointer
loaded from `0x9678 + %rdx*8`. If `%rdx` is `nullptr` (or a dangling
pointer), the read at `(%rdx + 0x1a)` is invalid.

The function signature (Windows x64 calling convention):
- arg 1 (`%rcx`): outer bound 1
- arg 2 (`%rdx`): outer bound 2 → accumulated
- arg 3 (`%r8`): table pointer → `0x9678 + idx*8` = some indexed table
- arg 4 (`%r9`): 1-byte field at offset 0x1a → houseID or itemID

## What likely matches this signature

`Map::isWithinBuildRange(int x, int y, const House* pHouse)` — scans
tiles around (x, y) within BUILDRANGE, for each tile reads the
**owner's houseID** at offset 0x1a of the ObjectManager entry, and
compares it to `pHouse->getHouseID()`.

## What it means for House_REBELS

If the ObjectManager has a stale entry (e.g. a HOUSE_REBELS object
whose internal pointer was nulled out during the previous mission), the
scan dereferences a stale pointer and crashes.
