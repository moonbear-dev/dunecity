# v1.0.269 RDP test results

## What we tried

Tried to reproduce the vanilla Windtrap crash (`0xC0000005 at BB9254C8`) by RDPing
into the Windows test box at 108.61.251.236, installing v1.0.269, and launching it
with diagnostic SDL_Log enabled.

## What happened

With `SDL_VIDEODRIVER=dummy` and `SDL_AUDIODRIVER=dummy`, the game gets through:

1. SDL_Init (audio + video + timer)
2. Audio init (28 channels allocated)
3. PAK file loading (all 19 PAK files)
4. Mod system init (vanilla mod active)
5. `Setting video mode...`
6. **`SDL_CreateRenderer failed: Couldn't find matching render driver`**

Then exits silently (no crash dump, no stack trace). The game can't render anything
under the dummy driver, so the vanilla Windtrap render-path crash is never reached.

## What this tells us

- The vanilla crash is **render-dependent** — it needs a real display backend
- My diagnostic SDL_Log patches in `ObjectBase::createObject`, `WindTrap::init`,
  `StructureBase::blitToScreen`, and `House::placeStructure` are all
  **upstream of `SDL_CreateRenderer`** and will fire if the game gets past this
  point with a real display
- The bug we need to reproduce is in the render path (blitToScreen,
  generateAdvancedWindtrapAnimationFrames, or generateWindtrapAnimationFrames)
  — diagnostic patches at those call sites should reveal it

## Diagnostic patches in v1.0.269

- `src/ObjectBase.cpp:createObject` — logs every itemID created via the
  scenario loader. Last log line before crash = trigger itemID.
- `src/structures/WindTrap.cpp:init` — logs houseID + graphic[0..2] pointers.
  Use to confirm `graphic[currentZoomlevel]` is valid at construction time.
- `src/structures/StructureBase.cpp:blitToScreen` — logs warning if
  `graphic[currentZoomlevel]` is null, skips blit. The actual crash is downstream
  of this check.
- `src/House.cpp:placeStructure` — logs every placement call. Last log line
  before crash = trigger placement.
