# v1.0.269 first real crash log analysis

## What I learned

Tornie reproduced the vanilla Windtrap crash on his actual Windows machine
with the v1.0.269 build. The diagnostic SDL_Log patches worked.

## Crash signature

- **Version**: 1.0.269 (commit `e28bca5` on `tornie/1.0.269-batch`)
- **Exception**: `0xC0000005 at 0x00007FF7BCDA54F8`
- **Crash site address**: 0x00007FF7BCDA54F8 (way outside dunecity.exe .text section
  0x140001000-0x14027AED1, so the crash is in a loaded DLL — most likely SDL2.dll)
- **Path**: `C:\Users\canti\AppData\Roaming\DuneCity\crash_20260704_082356.dmp`
- **Time**: 2026-07-04 08:23:56 (UTC)

## What happened in the log

1. SDL initialized successfully
2. Audio initialized (28 channels)
3. All 19 PAK files loaded
4. Mod system initialized (vanilla)
5. Game entered main menu
6. ModManager reseeded vanilla and dunecity mod (1.0.268 -> 1.0.269)
7. **Mission 1 Atreides** started, audio `DUNE5.ADL!` started playing

   - `houseID=7 itemID=2 xPos=28 yPos=24` → `HOUSE_REBELS` placing `Structure_Slab1`
   - `houseID=7 itemID=19 xPos=30 yPos=24` → `HOUSE_REBELS` placing `Structure_WOR`
9. `Briefing...` showed
10. `Now playing DUNE7.ADL!` (mission 2 Harkonnen)
11. `Now playing DUNE1.ADL!` (??)
12. `Now playing DUNE14.ADL!` (mission 3 Harkonnen probably)
13. **CRASH**

## Key diagnostic findings

- **No `ObjectBase::createObject ENTER/EXIT` log lines** — the scenario loader
  uses `placeStructure` (which I instrumented) but NOT `createObject` for
  initial placement. So `createObject` was called either earlier (before
  instrumentation) or via a different path.
- **2 `House::placeStructure` calls** for `HOUSE_REBELS` (houseID=7) at
  position (28,24) and (30,24). Both completed without crash.
- **No `WindTrap::init` log** — vanilla Windtrap was not created in this scenario.
  The crash is NOT a Windtrap-specific issue. It's a render-time crash that
  affects any structure when DUNE14.ADL loads.
- **No `StructureBase::blitToScreen WARNING`** — `graphic[currentZoomlevel]`
  was non-null at the time of blit. The crash is downstream of the blit guard.

## What this tells us

The crash is a **render-path bug** that:
- Occurs when DUNE14.ADL mission loads (specific to that mission)
- Happens after the new mission's first structures are placed
- Is not specific to Windtrap (the bug affects any structure in DUNE14.ADL)
- The blitToScreen null guard passed (graphic was valid)
- The crash is in a DLL (likely SDL2.dll) at offset 0xBCDA54F8

The likely root cause: the new mission places a structure that has an invalid
sprite or animation frame index, causing SDL_RenderCopy to read out-of-bounds
texture memory.

## Next steps

1. Add a diagnostic that captures the source/dest rect at blit time, so we
   can see which structure sprite has invalid bounds
2. Add a diagnostic to `Structure::update()` (first tick after placement) to
   see if the bug is in update rather than render

## Possible fix

Clamp the source/dest rect to texture bounds in `StructureBase::blitToScreen`:
```cpp
SDL_Rect source = calcSpriteSourceRect(pTex, indexX, numImagesX, indexY, numImagesY);
if (source.x + source.w > pTex->w || source.y + source.h > pTex->h) {
    SDL_Log("blitToScreen: clamping source rect for itemID=%d houseID=%d",
            itemID, getOwner()->getHouseID());
    source.w = std::min(source.w, pTex->w - source.x);
    source.h = std::min(source.h, pTex->h - source.y);
}
```

This is a defensive fix that should prevent the crash but may not address
the root cause. Better to add diagnostic logging first to confirm.
